## 2. 目录结构怎么理解

### 2.1 底层推理内核

- [src](/home/i_liuxinyu/llama.cpp-omni/src)
  这一层是 `libllama` 主体，负责模型、context、batch、KV cache、graph、sampling。

你可以把它映射成：

## 3. 用 vLLM 视角看这个项目

如果类比 vLLM：

- vLLM 的 engine / executor / kv cache / sampling
  对应这里的 [src](/home/i_liuxinyu/llama.cpp-omni/src)
- vLLM 依赖的 PyTorch + custom kernel backend
  对应这里的 [ggml](/home/i_liuxinyu/llama.cpp-omni/ggml)
- 你自己在 vLLM 外封装的多模态应用逻辑
  对应这里的 [tools/omni](/home/i_liuxinyu/llama.cpp-omni/tools/omni)

最大的风格差异是：

- vLLM 更像 “统一 serving runtime”
- llama.cpp-omni 更像 “底层推理库 + 若干 C++ CLI/应用层拼装”

## 6. 调用链拆解

### 6.1 启动阶段

`main`
-> `resolve_model_paths`
-> 组装 `common_params`
-> `omni_init`

`omni_init` 是整个 omni runtime 的总装函数。它负责：

- 创建 `omni_context`
- 加载或复用 LLM 模型
- 加载 TTS 模型
- 加载 APM 音频模型
- 初始化线程队列
- 初始化 Token2Wav 相关资源
- 缓存 special token
- 配置 simplex / duplex 模式

这一步类似 vLLM 里的：

- 载入模型
- 创建 engine/context
- 初始化 KV cache / sampling / runtime state

### 6.2 测试入口阶段

`test_case`
-> 循环调用 `stream_prefill(index=0..n-1)`
-> 所有输入 prefill 完成后调用 `stream_decode`

这里的意图很明确：

- `stream_prefill` 负责把多模态输入压进 LLM 上下文
- `stream_decode` 负责真正开始生成

在 vLLM 里你更习惯 `generate(prompt)` 一次包办；这里被拆成了“先手工 prefill，再单独 decode”。

### 6.3 `stream_prefill` 阶段

`stream_prefill` 是整个项目最值得精读的函数之一。

它按 `index` 分成两类：

#### 情况 A：`index == 0`

这一步主要不是用户输入，而是“系统前缀初始化”：

- prefill system prompt
- 如果配置了 voice cloning，会把 reference audio 的 embedding 一起 prefill
- 设置 `n_keep`
- 注册滑窗保护区
- 如果是 async 模式，会在这里启动工作线程

你可以把它理解成：

- 构造对话前缀
- 把不能被滑窗丢掉的系统区段固定下来
- 启动异步流水线

#### 情况 B：`index > 0`

这一阶段处理真实用户输入：

- 如果有图像，走 vision encoder，得到 vision embeddings
- 如果有音频，走 audio encoder，得到 audio embeddings
- 然后通过 `prefill_with_emb(...)` 把 embedding 喂进 LLM

同步模式下：

- `stream_prefill`
  -> `omni_image_embed_make_chunks_with_filename` / `omni_audio_embed_make_with_filename`
  -> `prefill_with_emb`
  -> `llama_decode`

异步模式下：

- `stream_prefill`
  -> 生成 `omni_embeds`
  -> push 到 `llm_thread_info->queue`
  -> `llm_thread_func` 消费它

### 6.4 `prefill_with_emb` 的位置

[omni.cpp](/home/i_liuxinyu/llama.cpp-omni/tools/omni/omni.cpp) 里的 `prefill_with_emb(...)` 很关键。

它本质上做的是：

- 准备 embedding batch
- 调 `llama_decode(...)`
- 更新 `n_past`

也就是说，在这个工程里，“prefill embedding” 和 “token decode” 最后都会落到 `llama_decode(...)` 这类底层调用上。

### 6.5 异步 LLM 线程

async 模式时，主链会变成：

`stream_prefill`
-> push `omni_embeds` 到 `llm_thread_info->queue`
-> `llm_thread_func`
-> 对 vision/audio embeddings 做 prefill
-> 触发后续 decode / 输出队列

`llm_thread_func(...)` 的职责是：

- 等待输入队列
- 从队列取出多模态 embedding
- 调 `prefill_with_emb(...)`
- 管理 `need_speek`、chunk 边界、回包数据

如果你从 vLLM 视角看，这一层像是：

- 一个简化版、业务定制的 scheduler + worker loop

但它不是做大规模请求调度，而是为 omni 实时流式场景服务。

### 6.6 `stream_decode` 阶段

`stream_decode(...)` 负责正式生成文本 / 音频 token。

你可以把它理解成：

- 从当前已经 prefill 好的 context 开始 autoregressive decode
- 识别特殊 token，例如 `<|speak|>`, `<|listen|>`, `<|chunk_eos|>` 等
- 把生成出来的 chunk 送到 TTS 队列

主逻辑上像这样：

`stream_decode`
-> 需要时启动 TTS / T2W 线程
-> 循环做 LLM decode
-> 解析 token 类型
-> 组装 `LLMOut`
-> push 到 `tts_thread_info->queue`

这个函数是“LLM 结果如何驱动语音输出”的核心中枢。

### 6.7 TTS 线程

生成后的文本 / hidden states 不直接变成 wav，而是先进入 TTS。

链路是：

`stream_decode`
-> push `LLMOut`
-> `tts_thread_func(...)` 或 `tts_thread_func_duplex(...)`
-> 生成 audio token
-> push `T2WOut`

这一步的关键特点是：

- TTS 不只是吃文本，有时还会结合 hidden states / token ids
- duplex 模式和 simplex 模式用了不同线程函数

### 6.8 Token2Wav 线程

最后一段是 vocoder：

`tts_thread_func*`
-> push `T2WOut`
-> `t2w_thread_func(...)`
-> 实际走 Python service 或 C++ 实现
-> 输出 wav 文件

从源码看，当前主实现明显偏向 Python Token2Wav 服务：

- 启动 Python 子进程
- 通过 stdin/stdout 发送 JSON 命令
- process token chunk
- 滑窗式拼接音频

也就是说，虽然主框架是 C++，但最后的 Token2Wav 不是完全内嵌进 `libllama` 的，而是一个外部服务式子流程。

## 7. 线程与队列关系

从 [omni.h](/home/i_liuxinyu/llama.cpp-omni/tools/omni/omni.h) 看，`omni_context` 里有三类线程和三类队列：

- `llm_thread` / `llm_thread_info`
- `tts_thread` / `tts_thread_info`
- `t2w_thread` / `t2w_thread_info`

它们之间的数据流是：

输入媒体
-> `stream_prefill`
-> `llm_thread`
-> `stream_decode`
-> `tts_thread`
-> `t2w_thread`
-> wav 文件

这是这个项目和普通 `llama.cpp` 最大的区别之一：它不是单线程“给个 prompt 出个文本”，而是一个小型流水线 runtime。

## 9. 一句话总结

这个项目不是“另一个 vLLM”，而是：

- 用 `llama.cpp + ggml` 作为底层推理内核
- 在上面叠了一层专门面向 `MiniCPM-o` 的多线程 omni 应用 runtime

如果你熟悉 vLLM，最容易上手的切入点不是先看 `ggml`，而是先把 [tools/omni/omni-cli.cpp](/home/i_liuxinyu/llama.cpp-omni/tools/omni/omni-cli.cpp) -> [tools/omni/omni.cpp](/home/i_liuxinyu/llama.cpp-omni/tools/omni/omni.cpp) 这条主执行链走通。
